Clone VMM

The VM engine.

A KVM-based virtual machine monitor built for high-density workloads. Boot once, fork many. Every fork shares the same physical memory pages until it writes. Real hardware isolation, not containers.

Instant Cloning

Snapshot a running VM, then spawn clones in under 20ms. Full 4GB Ubuntu with networking in ~160ms. No kernel boot. Memory is shared automatically — 2 cloned 4GB VMs use ~1GB host RAM.

Three-Layer Memory

Overcommit (pages allocated on first write) + KSM (deduplicate identical pages) + Balloon (reclaim idle memory). 10 idle 512MB VMs use ~200MB of host RAM.

Live Migration

Pre-copy migration over TCP. VM keeps running while memory transfers. Final switchover in ~1ms. Zero-downtime host maintenance.

virtio-fs

Share host directories into guests without an external daemon. Mount user home directories, inject configuration files, share templates.

GPU Passthrough

VFIO device passthrough for GPUs and other PCI devices. Run ML inference inside a VM with direct hardware access.

Single Binary

25K lines of Rust. No libvirt, no QEMU, no forked codebases. Boots VMs, manages devices, handles migration. One binary.

Army

The orchestrator.

Army manages VMs across hosts via a REST API on top of Clone's fast fork. Multi-host scheduling, quota enforcement, SQLite persistence. Single binary, zero external dependencies beyond Clone.

# Start the orchestrator army serve --addr :8080 --db army.db # Start an agent on each compute host sudo army agent --addr :9090 --clone-socket /run/clone/control.sock # Create a shell in <20ms army shell create --template shell-base --tenant default # Execute a command army vm exec vm-3a8f -- uname -a Linux clone 6.5.0-35-generic x86_64

Multi-Host

Army agents run on each compute host. The central server schedules VMs across them. Add a box, register it, start serving.

Quotas

Per-tenant limits on VMs, memory, and vCPUs. Enforced on every create. Built-in multi-tenancy.

Reconciliation

On startup, Army queries each Clone daemon and reconciles stored state. Orphaned VMs get cleaned up. Survives crashes.

Use cases

What people build with it.

Clone and Army are general-purpose VM infrastructure. If you need many isolated instances of the same thing, this is the cheapest way to run them.

Shell Hosting

Give every user their own Linux shell with KVM isolation. Pre-install everything, share the base automatically. 1,000 users on one box. The original use case.

AI Agent Swarms

Spin up hundreds of isolated agents, each in their own VM. Fork from a template with the model and tools pre-loaded. Agents can't interfere with each other.

CI/CD Runners

Fresh VM per build, cloned in under 20ms. Shared base image with your toolchain warm. Destroy after the build. Faster and cheaper than containers.

Sandboxed Code Execution

Run untrusted code in a real VM, not a container. Each submission gets its own KVM-isolated environment. Fork, execute, destroy. LeetCode-style at scale.

Game Servers

One template, hundreds of instances. Each server isolated. Memory sharing means the game binary is loaded once no matter how many servers you run.

Bot Army

Yes, you can raise an army of bots. Each one in its own VM, its own IP, its own filesystem. Fork from a template, deploy your bot code, scale to hundreds. We named the orchestrator Army for a reason.

At scale

Why infrastructure companies want this.

If you run VMs for customers, memory sharing changes your unit economics. The more similar your workloads, the bigger the savings.

Cloud Shell Providers

AWS CloudShell, GCP Cloud Shell, Azure Cloud Shell — all give users an ephemeral Linux environment. With Clone, 1,000 shells share one base image in memory. 40x density improvement.

Dev Environment Platforms

GitHub Codespaces, Gitpod, Replit — all provision isolated dev environments. With Clone, the base image is shared across every instance. A $200/mo server replaces $10,000/mo in traditional VM infrastructure at 1,000 users.

FaaS / Serverless

AWS Lambda, Cloudflare Workers, Fly Machines — all need fast cold starts. Clone spawns from a warm template in under 20ms with a standard kernel. No custom kernels required.

Edge Compute

Fly.io, Railway, Render — run customer workloads on shared infrastructure. With automatic memory sharing, the common base costs zero per additional instance. Run 10x more apps per edge node.

Shared Hosting (Modern)

The original shared hosting model died because chroot isolation wasn't secure. Clone brings it back with KVM hardware isolation and automatic memory sharing. The density of shared hosting with the security of dedicated VMs.

AI Inference

Load a model into a template VM with GPU passthrough. Fork instances that share the model weights in memory automatically. Model weights loaded once, served many.

The math

Traditional VMs: 1,000 instances × 1GB each = 1TB RAM needed.
Clone: 1,000 instances × ~50MB private pages + 600MB shared base = ~51GB RAM.
That's 20x less infrastructure for the same workload. At cloud prices, that's the difference between $50,000/mo and $2,500/mo in compute.
